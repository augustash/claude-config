---
name: Cron off-path page_cache re-prime (kill a reindex/purge-driven stampede)
description: A high-traffic page that's uncacheable at dynamic_page_cache (a CSRF/Turnstile/webform bubbles max-age 0) survives only on anonymous page_cache. When a cron periodically invalidates it, the cold window dogpiles (page_cache does not coalesce). Fix: cron renders each high-volume variant off-path and atomically overwrites its page_cache entry under the canonical core cid — never deleting it, so there is never a cold hole.
type: reference
---

**When it applies.** A page is **uncacheable by Drupal's `dynamic_page_cache`** because it carries a form whose CSRF token (or a Turnstile/captcha) bubbles `max-age 0` — so the only thing serving it fast to anonymous traffic is the **`page_cache` (full-HTML) bin**. If a cron job periodically invalidates that entry (a Search API reindex, an entity resave, a cache-tag flush — anything that clears the page's tag every N minutes), each clear opens a cold window. And **`page_cache` does not coalesce**: N concurrent requests into a cold entry each render the page in full. With a small php-fpm pool the burst overcommits the workers → stampede. Diagnose it as burst-not-load: the periodicity matches the cron interval, and the slow stack frames are usually cold *metadata* rebuild (Views/Search-API data, plugin/field defs), not the page's own rows.

Tracking-param strips, edge cache rules, and the deploy-time Quicksilver warmer ([pantheon-quicksilver-cache-warmer.md](pantheon-quicksilver-cache-warmer.md)) don't touch this — that warmer explicitly punts the "mid-day cron purge collides with a burst" case to app level. **This pattern is that app-level answer.**

**The fix.** In the cron job, after the data change, re-prime the `page_cache` entry off the request path for each high-volume variant — and crucially, **overwrite in place, never invalidate**:

1. **Render off-path via loopback curl.** `https://127.0.0.1<path>` + a `Host: <public-host>` header + `-k` + a sentinel query param (e.g. `_prime=1`) to bust the existing HIT and force a fresh render; capture the 200 HTML and rebuild it as an `HtmlResponse`. The loopback curl is required because it goes through the **full middleware stack**, so page_cache's `StackMiddleware` actually stores the result. An internal `\Drupal::service('http_kernel')->handle($request)` does **not** — it bypasses the page_cache middleware, so it renders but never populates the bin. (Hard-won: the internal-subrequest approach silently no-ops.)
2. **Write under the canonical cid, TAGLESS.** `\Drupal::cache('page')->set($cid, $response, CacheBackendInterface::CACHE_PERMANENT)`. The cid is core `PageCache::getCacheId()`: `$request->getSchemeAndHttpHost() . $request->getRequestUri() . ':' . $format`, and `$format` is **empty** for HTML pages — so the cid ends in a bare `:` (e.g. `https://www.example.com/path:`). Tagless because the page's freshness tag is the *same* tag the cron invalidates: carry it and the next cron cycle deletes your freshly-primed entry, re-opening the hole. Instead cron is the **sole owner** — freshness comes from re-rendering every cycle, not tag invalidation. Trade-off: an editorial chrome change (title/menu) appears on the next cron cycle, not instantly — in-band if the cron is frequent.
3. **Never delete.** The old entry keeps serving (~tens of ms) until the single `set()` overwrites it in place — an atomic blue-green swap. Traffic only ever experiences a Redis SET (old value → new value), never a miss. This is the whole point: do **not** invalidate-then-reprime (that opens the very window you're closing); overwrite.

**Gotchas that cost real time:**
- **Public host, per-env.** The cid uses the host the *request* arrived on. Live often redirects all traffic to a primary domain (`www.…`), so that's the live cid host — but cron runs as CLI and never sees that redirect. Resolve it explicitly (`live` env → a primary-domain constant; else `$GLOBALS['base_url']`), or you write entries under a host no real request reads.
- **Enumerate every variant a real visitor reads.** One cid per canonical URI × per configured language (language-prefixed paths for non-default langs). Miss one and that variant stays permanently cold. Watch query-param variants that render identically but key separately — e.g. a default-tab URL `?type=arrival` vs the bare path are *different* cids.
- **Drop any explicit page invalidation** (`invalidateTags(['node:NNN'])`) the cron used to do — it now fights the re-prime.
- **`page_cache` becomes a hard module dependency** (the `@cache.page` service); declare it in info.yml.

**Proving it / regression coverage.** Override the loopback-render seam and the rest is pure → kernel-testable: assert the variant enumeration (which cids), and the reprime behaviour (writes every variant; overwrites in place; a non-200 render writes nothing; one variant throwing doesn't stop the others). The never-a-cold-hole property is showable directly too — fire N concurrent at the page *during* a re-prime and confirm every request HITs (old then new), zero renders. Drive that as a *repeated burst* (re-cold, fire N, measure those N, repeat), not a sustained load test — a sustained test re-warms after one request and can't see the stampede.
